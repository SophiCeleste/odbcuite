# SDD Report — effective_to dtype fix (scd2_load_duckdb / scd2_load_azure)

## Bug

`scd2_load_duckdb` and `scd2_load_azure` in `ns_utils.py` set the new-version
DataFrame's `effective_to` column with a bare Python `None`:

```python
to_insert["effective_to"]   = None
```

On the first run this produces an all-`None` column. DuckDB's
`CREATE TABLE ... AS SELECT * FROM to_insert` infers that column as `INTEGER`.
A later CLOSE `UPDATE ... SET effective_to = <date>` then fails with:

```
_duckdb.ConversionException: Conversion Error: Unimplemented type for cast (DATE -> INTEGER)
```

The identical latent typing problem exists in `scd2_load_azure` (same
`= None` assignment).

## Concerns

- **`scd2_load_azure` fix is unverified by automated test.** No live SQL
  Server is available in this environment. The fix mirrors the DuckDB fix
  exactly (identical line, identical reasoning: an all-NaT
  `datetime64[ns]` Series will be written by `to_sql` as a proper
  DATETIME/DATETIME2 column instead of an all-None object column that
  pandas/SQLAlchemy might otherwise coerce inconsistently). This is
  covered by analogy only — recommend a smoke test against a real Azure
  SQL Server table before relying on this path in production if it hasn't
  been exercised recently.
